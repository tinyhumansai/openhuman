# Meet Agent

The Meet Agent lets OpenHuman join a Google Meet call as an independent participant — a second, hidden browser window that the desktop app controls end-to-end. Stage 1 (this PR) proves the fundamentals: cookie-sharing, auto-join, and lifecycle events. Rendering, audio, and the LLM loop come in later stages.

## What Stage 1 ships

- A hidden webview (`1280x720`, positioned off-screen at `-4000,-4000`) that shares the user's existing Google login via a shared cookie store.
- Auto-join with mic and camera forced off before clicking the join button.
- Lifecycle events surfaced to the React layer: `meet_agent_joined`, `meet_agent_left`, `meet_agent_failed`.
- A Leave command that clicks the Leave-call button before closing the webview.
- A staging-only dev panel on the `/human` tab for quick manual testing.

**Not in Stage 1:** no avatar rendering, no TTS/audio output, no STT/transcription input, no LLM decision loop, no multi-participant awareness.

## Architecture

### Sibling webview with shared cookies

`webview_meet_agent_join` (Tauri command) calls `data_directory_for(app, account_id)` — the same function used by the user-facing google-meet webview. Both webviews share `{app_local_data_dir}/webview_accounts/{account_id}`, so Google session cookies are automatically shared. No re-authentication is needed as long as the user is already signed in via the regular google-meet channel.

The agent webview uses a different label (`acct_{id}_agent`) so it stays independent and can be closed without touching the user-facing view.

### Role-gated injection

`build_agent_init_script` injects:

```
window.__OPENHUMAN_RECIPE_CTX__ = { accountId, provider: "google-meet", role: "agent", meetingUrl };
RUNTIME_JS   // provides window.__openhumanRecipe (emit / log / loop)
MEET_AGENT_JS // the auto-join polling loop
```

`agent.js` bails immediately if `role !== "agent"`, so the same binary can be injected into both webview types without risk. The user-facing `recipe.js` is never injected into the agent webview.

### Navigation guard

The agent webview's `on_navigation` callback allows only `meet.google.com`, `accounts.google.com`, and `www.google.com`. All other hosts are blocked so the agent cannot navigate outside Meet even if a link is programmatically triggered.

## DOM selectors (stable-ish)

`agent.js` depends on these DOM patterns. Meet's front-end engineers rename selectors roughly once every few months during large refactors — the same maintenance cadence as `recipe.js`.

| Selector | Purpose |
|---|---|
| `button[jsname="Qx7uuf"]` | Pre-join primary action (Join / Ask to join) |
| `button[aria-label="Join now\|Ask to join\|Join"]` | Fallback for join button |
| `[role="button"][aria-label*="microphone" i]` | Pre-join mic toggle |
| `[role="button"][aria-label*="camera" i]` | Pre-join camera toggle |
| `[data-self-name]`, `[data-participant-id]` | In-call presence signals |
| `[aria-label="Leave call"]` | Leave-call button |
| `[jsname="CQylAd"]` | Leave button jsname fallback |

When a selector stops working, check DevTools on the Meet lobby/in-call page and update the relevant helper in `agent.js`. All helpers are pure functions tested in `app/test/meet-agent.test.ts`.

## Trying it from the staging UI

1. Make sure you have a google-meet account connected in OpenHuman (Settings → Channels → Google Meet) and are signed in.
2. Navigate to the `/human` tab — only available in staging/development builds.
3. At the bottom-left you will see the **Meet Agent (staging)** panel.
4. Enter the `account-id` (the account ID shown in Settings for your Meet account).
5. Enter a `https://meet.google.com/xxx-xxxx-xxx` URL.
6. Click **Join**. The agent webview spawns off-screen and attempts to join.
7. Watch the status line: it transitions from `joining...` → `joined xxx-xxxx-xxx at HH:MM:SS` once the agent enters the call.
8. Click **Leave** to gracefully exit or close the app entirely.

Note: you cannot observe the agent webview visually in Stage 1 — it is intentionally hidden. Check DevTools (`View → Developer → Open Web Inspector for Webview`) to inspect it if needed.

## Roadmap

### Stage 2 — Camera: mascot avatar

Render the mascot (`Ghosty`) into a `<canvas>` element inside the agent webview and use `getDisplayMedia`/`captureStream` to pipe the canvas as the agent's camera feed. The meeting sees the mascot's face instead of a blank tile.

### Stage 3 — Mic: TTS output

Route TTS audio (generated by the Rust core's speech pipeline or an external provider) into the agent's virtual microphone so it can speak in the call. Requires a virtual audio device or a WebAudio-based output pipeline.

### Stage 4 — Incoming audio: STT

Capture the meeting's incoming audio track inside the agent webview, stream raw PCM to the Rust core for Whisper transcription, and emit structured `meet_transcript_chunk` events.

### Stage 5 — LLM loop

Wire the transcript stream into the OpenHuman agent loop: inbound speech → STT → context window → LLM → TTS → outbound audio. The agent becomes a full participant that can answer questions, summarise, and take notes.
